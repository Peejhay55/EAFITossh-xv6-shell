# Documentación Técnica: Lazy Allocation Real

En esta etapa se completó la implementación de la asignación diferida de memoria, permitiendo que el sistema sea funcional y asigne recursos físicos solo cuando son realmente necesarios.

## 1. Implementación en `usertrap()`

Se modificó el manejador de excepciones en [kernel/trap.c](kernel/trap.c) para procesar los *Page Faults* de forma inteligente en lugar de simplemente terminar el proceso.

### Lógica del Manejador:
1. **Captura de la dirección**: Se obtiene la dirección virtual que causó la falla desde el registro `stval`.
2. **Validación de rango**: Se comprueba si la dirección es "legal", es decir, si está por debajo del límite de memoria virtual del proceso (`p->sz`).
3. **Asignación Física**:
   - Si la dirección es legal, se invoca a `vmfault()`.
   - `vmfault()` realiza un `kalloc()` (obtiene una página física), la inicializa en 0 (`memset`) y la mapea en la tabla de páginas del proceso con los permisos adecuados (`PTE_W|PTE_U|PTE_R`).
4. **Continuación**: Si la asignación tiene éxito, el kernel retorna al espacio de usuario. Como no se llamó a `setkilled()`, el procesador reintenta la instrucción que falló, esta vez con éxito.
5. **Fallo Real**: Si la dirección es ilegal (fuera de `p->sz`), se mantiene el comportamiento de imprimir el error y terminar el proceso.

## 2. Beneficios Observados

- **Estabilidad del Sistema**: La shell y el comando `EAFITossh` vuelven a funcionar correctamente, ya que sus solicitudes de memoria vía `sbrk/malloc` se resuelven bajo demanda.
- **Eficiencia**: El sistema ya no malgasta memoria física en páginas que los procesos solicitan pero nunca llegan a utilizar.

## 3. Pruebas Realizadas

Se utilizó el programa `tlazy`:
- **Antes**: El programa moría al intentar acceder a la memoria.
- **Ahora**: El programa completa su ejecución correctamente. El kernel gestiona el fallo de forma transparente para el usuario.
