# Documentación Técnica: Detección de Page Faults en xv6

En este proyecto se implementó la detección y reporte de errores de página (*Page Faults*) en el kernel de xv6-riscv, junto con una herramienta en el espacio de usuario para validar el comportamiento.

## 1. Modificaciones en el Kernel

Se intervino el archivo [kernel/trap.c](kernel/trap.c) dentro de la función `usertrap()`. Esta función es la encargada de manejar todas las excepciones e interrupciones que ocurren mientras un proceso de usuario está en ejecución.

### Detección de Excepciones
Se añadieron condiciones para identificar específicamente dos tipos de fallas definidas por la arquitectura RISC-V:
- **Load Page Fault (scause 13):** Ocurre cuando un proceso intenta leer desde una dirección virtual no mapeada o sin permisos de lectura.
- **Store Page Fault (scause 15):** Ocurre cuando un proceso intenta escribir en una dirección virtual no mapeada o sin permisos de escritura.

### Reporte y Manejo
Cuando se detecta una de estas excepciones, el kernel realiza lo siguiente:
1. Extrae el PID del proceso actual (`p->pid`).
2. Obtiene el código de la causa de la excepción mediante `r_scause()`.
3. Obtiene la dirección virtual que originó la falla mediante `r_stval()`.
4. Imprime un mensaje informativo en la consola del kernel:
   `page fault: pid=X scause=Y stval=0xZ`
5. Marca el proceso para ser terminado mediante `setkilled(p)`.

## 2. Herramienta de Prueba (User Space)

Se creó el programa [user/tpf.c](user/tpf.c) para validar la implementación. Este programa permite inducir fallos de manera controlada:

- **tpf load:** Intenta acceder a la dirección `0x4000` para lectura.
- **tpf store:** Intenta escribir el valor `1` en la dirección `0x8000`.

Ambas direcciones son arbitrarias y no están mapeadas en el espacio de direcciones del proceso de usuario (excepto si crecieran mediante `sbrk`, lo cual no ocurre en esta prueba), garantizando el disparo del *Page Fault*.

## 3. Integración en el Sistema

Se modificó el [Makefile](Makefile) para incluir el binario `_tpf` en la variable `UPROGS`. Esto asegura que el programa sea compilado y cargado en el sistema de archivos (`fs.img`) al construir el proyecto.

## 4. Guía de uso (Paso a paso)

Para verificar la implementación, siga los siguientes pasos:

1. **Compilar y ejecutar el sistema**:
   En la raíz del proyecto, ejecute el comando:
   ```bash
   make qemu
   ```

2. **Probar Load Page Fault (Lectura inválida)**:
   Dentro de la shell de xv6, ejecute el comando:
   ```bash
   tpf load
   ```
   *   **Resultado esperado**: El kernel imprimirá `page fault: pid=X scause=13 stval=0x4000` y el proceso será terminado.

3. **Probar Store Page Fault (Escritura inválida)**:
   Nuevamente en la shell de xv6, ejecute el comando:
   ```bash
   tpf store
   ```
   *   **Resultado esperado**: El kernel imprimirá `page fault: pid=X scause=15 stval=0x8000` y el proceso será terminado.

4. **Verificar el PID y las Direcciones**:
   Observe que el PID cambia según el orden de ejecución y que `stval` coincide exactamente con las direcciones definidas en el programa de prueba (`0x4000` para lectura y `0x8000` para escritura).
