# Documentación Técnica: Introducción a Lazy Allocation

En este punto se transformó la llamada al sistema `sbrk()` para implementar la asignación diferida (*Lazy Allocation*).

## 1. Modificación de `sys_sbrk()`

Originalmente, `sbrk(n)` llamaba a `growproc(n)`, lo cual asignaba inmediatamente páginas físicas de memoria mediante `kalloc()` y las mapeaba en la tabla de páginas del proceso.

### Cambios realizados en [kernel/sysproc.c](kernel/sysproc.c):
- Se eliminó la llamada a `growproc(n)` cuando el incremento `n` es positivo.
- En su lugar, simplemente se incrementa el tamaño virtual del proceso (`myproc()->sz += n`).
- Para casos donde `n` es negativo (liberación de memoria), se mantuvo la llamada a `growproc(n)` para liberar los recursos físicos de inmediato.

Con este cambio, el proceso "cree" que tiene más memoria, pero no hay páginas físicas reales respaldando esas nuevas direcciones virtuales.

## 2. Comportamiento Esperado

Al ser una asignación puramente virtual:
1. El proceso intenta acceder a una dirección dentro del nuevo rango.
2. La MMU (Memory Management Unit) no encuentra el mapeo en la tabla de páginas.
3. Se dispara un **Store Page Fault** (causa 15) o **Load Page Fault** (causa 13).
4. El kernel captura la excepción (gracias a la implementación del Punto 1) y termina el proceso.

## 3. Programa de Prueba: `tlazy`

Se implementó [user/tlazy.c](user/tlazy.c) con la siguiente lógica:
1. Llama a `sbrk(4096)`.
2. Imprime la dirección recibida.
3. Intenta escribir un carácter en esa dirección (`*p = 'a'`).

### Resultado de la prueba:
Al ejecutar `tlazy` en la shell:
```bash
tlazy: Iniciando prueba de Lazy Allocation...
tlazy: Memoria reservada virtualmente en 0x4000. Accediendo...
page fault: pid=X scause=15 stval=0x4000
```
El proceso es terminado correctamente por el kernel al detectar el acceso a una página aún no asignada físicamente.
