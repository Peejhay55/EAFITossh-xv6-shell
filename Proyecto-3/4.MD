# Documentación Técnica: Control y Visualización de Lazy Allocation

En este punto se instrumentó el kernel para realizar un seguimiento dinámico del comportamiento de la asignación diferida.

## 1. Instrumentación: Contador de Page Faults

Se añadió un mecanismo para cuantificar cuántas veces el kernel interviene para asignar memoria bajo demanda.

### Cambios en el Kernel:
- **`struct proc` ([kernel/proc.h](kernel/proc.h))**: Se agregó el campo `int pf_count` a la estructura del proceso.
- **`allocproc()` ([kernel/proc.c](kernel/proc.c))**: El contador se inicializa en 0 cada vez que se crea un proceso nuevo.
- **`usertrap()` ([kernel/trap.c](kernel/trap.c))**: 
  - Se incrementa `p->pf_count` cada vez que ocurre un Page Fault válido que es resuelto con éxito por `vmfault()`.
  - Se añadió un mensaje de log que muestra el progreso: `page fault: pid=X pf_count=Y stval=0xZ`.

## 2. Análisis del Comportamiento Dinámico

Se actualizó [user/tlazy.c](user/tlazy.c) para realizar pruebas de acceso a gran escala (10 páginas).

### Modos de Prueba:

1. **Acceso Secuencial (`tlazy`)**:
   - El programa accede a las páginas en orden (0, 1, 2...).
   - El kernel registra un incremento lineal en `pf_count`.

2. **Acceso Disperso/Aleatorio (`tlazy random`)**:
   - El programa accede a las páginas en orden inverso o salteado.
   - El kernel responde asignando las páginas solo según el orden de acceso real, demostrando la flexibilidad de la asignación diferida.

## 3. Resultado Esperado

Al ejecutar `tlazy`, se observará en la consola del kernel cómo el contador aumenta de 1 hasta 10 conforme el programa toca cada una de las 10 páginas solicitadas vía `sbrk`. Esto confirma que las páginas no se asignaron al inicio, sino exactamente en el momento de su primer uso.
